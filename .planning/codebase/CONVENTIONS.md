# Coding Conventions

**Analysis Date:** 2026-04-01
**Status:** Planned — no source code exists yet. All conventions are prescribed based on the chosen tech stack and project constraints.

## Naming Patterns

**Files:**
- Route files follow TanStack Start file-based routing: `app/routes/index.tsx`, `app/routes/tasks/$taskId.tsx`
- Server functions co-located with the route that uses them, or in `app/server/` for shared logic
- Component files: PascalCase — `TaskCard.tsx`, `TaskForm.tsx`
- Utility/helper files: camelCase — `formatDate.ts`, `parsePhoneNumber.ts`
- Type definition files: camelCase with `.types.ts` suffix — `task.types.ts`, `auth.types.ts`
- Supabase client and generated types: `app/lib/supabase.ts`, `app/lib/database.types.ts`

**Functions:**
- React components: PascalCase — `function TaskBoard() {}`
- Server functions (TanStack Start `createServerFn`): camelCase, verb-noun — `createTask`, `updateTaskStatus`, `processScreenshot`
- Event handlers: camelCase, `handle` prefix — `handleSubmit`, `handleStatusChange`
- Utility functions: camelCase, verb-noun — `formatRelativeTime`, `extractPhoneDigits`

**Variables:**
- camelCase throughout — `taskList`, `isLoading`, `currentUser`
- Boolean variables: `is`/`has`/`can` prefix — `isAuthenticated`, `hasOpenTasks`, `canDelete`
- Constants: SCREAMING_SNAKE_CASE for module-level constants — `MAX_UPLOAD_SIZE_BYTES`, `TASK_STATUS`

**Types and Interfaces:**
- PascalCase for all — `Task`, `StaffMember`, `RequestType`, `TaskStatus`
- Prefer `type` aliases over `interface` for plain data shapes
- Use `interface` only when extension is intended
- Database row types generated by Supabase CLI: `Database['public']['Tables']['tasks']['Row']`
- Prefer named aliases: `type Task = Database['public']['Tables']['tasks']['Row']`

**Enums:**
- Use `const` object + `as const` instead of TypeScript `enum`:
  ```typescript
  export const TASK_STATUS = {
    OPEN: 'open',
    IN_PROGRESS: 'in_progress',
    DONE: 'done',
  } as const;
  export type TaskStatus = typeof TASK_STATUS[keyof typeof TASK_STATUS];
  ```

## Code Style

**Formatting:**
- Prettier for all formatting (TypeScript, TSX, CSS)
- Config: `.prettierrc` at project root
- Recommended settings: single quotes, trailing commas (`'all'`), 100-character print width, 2-space indent
- Format on save enforced in editor config (`.vscode/settings.json`)

**Linting:**
- ESLint with TypeScript plugin (`@typescript-eslint/eslint-plugin`)
- `eslint-plugin-react-hooks` for hook dependency rules
- No unused variables (`@typescript-eslint/no-unused-vars` as error)
- No explicit `any` (`@typescript-eslint/no-explicit-any` as error — use `unknown` and narrow)
- `eslint-plugin-jsx-a11y` for accessibility (tab-accessible UI is required for desktop staff)

**TypeScript:**
- `strict: true` in `tsconfig.json` — no exceptions
- No `as` type assertions except when interfacing with untyped external data (and always with a comment)
- Supabase client typed with generated database types: `createClient<Database>(...)`
- All server function return types explicitly declared

## Import Organization

**Order (enforced by ESLint `import/order` or manual):**
1. Node built-ins (rare in this stack)
2. External packages — `react`, `@tanstack/start`, `@supabase/supabase-js`
3. Internal path aliases — `~/lib/supabase`, `~/components/TaskCard`
4. Relative imports — `./utils`, `../types`
5. Type-only imports last within each group — `import type { Task } from '~/lib/database.types'`

**Path Aliases:**
- `~` maps to `app/` — configure in `tsconfig.json` `paths` and Vite `resolve.alias`
- Use `~/` for all non-relative imports: `~/components/TaskCard`, `~/lib/supabase`, `~/server/tasks`

## TanStack Start Patterns

**File-Based Routing:**
- Routes live in `app/routes/`
- Layout routes use `_layout` prefix: `app/routes/_auth.tsx` for protected layout
- Dynamic segments: `app/routes/tasks/$taskId.tsx`
- Index routes: `app/routes/index.tsx`

**Server Functions:**
- Use `createServerFn` for all mutations and sensitive reads (auth-dependent queries, Claude API calls)
- Server functions live either co-located in route files or extracted to `app/server/`
- Always validate input with a schema (Zod) before processing
- Never call the Claude API or expose Supabase service-role key from client code — always via `createServerFn`
- Pattern:
  ```typescript
  import { createServerFn } from '@tanstack/start';
  import { z } from 'zod';

  const createTaskSchema = z.object({ ... });

  export const createTask = createServerFn('POST', async (input: unknown) => {
    const validated = createTaskSchema.parse(input);
    // supabase server client here
  });
  ```

**Loaders:**
- Use `loader` in route files for initial data fetching
- Loaders run server-side; use the Supabase server client (service role or session-scoped)
- Return typed data — define a `LoaderData` type per route

## Supabase Patterns

**Client Setup:**
- Browser client: `app/lib/supabase.client.ts` — `createBrowserClient<Database>`
- Server client: `app/lib/supabase.server.ts` — `createServerClient<Database>` (used in server functions and loaders)
- Never import the server client in client-side component code

**Row-Level Security:**
- All tables have RLS enabled — no exceptions
- `tasks` table: authenticated users can read all rows, insert/update/delete their own (or all staff — flat team)
- `push_subscriptions` table: users can only read/write their own subscription
- RLS policies defined in Supabase migrations, committed to `supabase/migrations/`

**Realtime:**
- Subscribe to Supabase Realtime channels in a React `useEffect` with proper cleanup
- Unsubscribe on component unmount to prevent memory leaks
- Channel naming: `tasks:all` for the shared board subscription
- Pattern:
  ```typescript
  useEffect(() => {
    const channel = supabase
      .channel('tasks:all')
      .on('postgres_changes', { event: '*', schema: 'public', table: 'tasks' }, (payload) => {
        // update local state
      })
      .subscribe();
    return () => { supabase.removeChannel(channel); };
  }, []);
  ```

**Queries:**
- Always use `.select()` with explicit column lists — no `select('*')` in production code
- Always handle the `error` return from Supabase queries — throw or surface to the user
- Type query results — use `data` typed via generated database types

## Styling Patterns (Tailwind CSS)

**Approach:**
- Utility-first Tailwind throughout — no custom CSS files except for global resets in `app/styles/globals.css`
- No CSS modules or styled-components
- `tailwind.config.ts` defines design tokens (colors, spacing) matching the app's needs

**Mobile-First:**
- All layout styles start at the smallest breakpoint (375px) and scale up
- Use `sm:`, `md:`, `lg:` prefixes for progressive enhancement
- Touch targets minimum 44x44px (iOS HIG guideline) — use `min-h-[44px] min-w-[44px]` on interactive elements
- Avoid hover-only states for critical interactions — iOS Safari has no hover

**Component Patterns:**
- Extract repeated utility combinations into small components, not custom CSS classes
- Use `cn()` (clsx + tailwind-merge) utility for conditional class merging:
  ```typescript
  import { cn } from '~/lib/utils';
  <div className={cn('base-classes', isActive && 'active-class')} />
  ```

**Color and Theming:**
- Define semantic color tokens in `tailwind.config.ts` (e.g., `task-open`, `task-done`)
- Avoid hardcoded hex values in class names

## Error Handling

**Patterns:**
- Server functions throw typed errors that propagate to the UI as TanStack Start error boundaries
- Define a shared `AppError` type for expected errors (validation, not-found, unauthorized)
- Unexpected errors (network failures, Supabase outages) caught at route error boundary — show user-friendly message
- Claude API failures (AI-05) fall back gracefully to the manual form — never block the user from creating a task manually
- Log errors server-side; never expose stack traces to the client

**Form Validation:**
- Validate with Zod on both client (for instant feedback) and server (for security)
- Surface field-level errors next to the relevant input, not just a top-level banner
- Required fields: client name, phone number (TASK-08)

## Logging

**Client:**
- No `console.log` in committed code — use `console.error` only for caught errors
- Remove all debug logging before commit

**Server:**
- Structured logging via `console.log` to stdout (Vercel captures this)
- Log shape: `{ level, event, taskId?, userId?, error? }` — always include relevant IDs

## Comments

**When to Comment:**
- Business logic that isn't obvious from the code — e.g., why a specific Claude prompt structure was chosen
- RLS policy intent (in migration files)
- Any `as` type assertion — explain why it's safe

**JSDoc:**
- Not required for internal utilities
- Add JSDoc to exported server functions and shared utilities that will be used across multiple routes

## iOS Safari Compatibility

**Critical constraints** (from PWA-01, PWA-03):
- No `position: fixed` on elements inside scrollable containers — causes iOS Safari layout bugs
- Use `-webkit-overflow-scrolling: touch` where applicable or Tailwind's `overflow-y-scroll` with `overscroll-behavior`
- Test file upload (`<input type="file">`) on iOS Safari — the `accept` attribute behavior differs
- PWA `display: standalone` — remove browser UI from consideration; handle safe areas with `env(safe-area-inset-*)`
- No `:hover` as the only interaction affordance — tap is the primary gesture
