# Architecture

**Analysis Date:** 2026-04-01
**Status:** PLANNED — no source code exists yet. This document describes intended architecture based on planning documents.

## System Overview

Massage Shop Task Manager is a PWA task board for a 3-5 person massage shop team. Staff log incoming call leads as tasks, the whole team sees the board in real time, and push notifications alert all devices when new tasks land. A Claude vision AI feature pre-fills task forms from KakaoTalk screenshots.

**Scale:** Single shop, 3-5 concurrent users. Simplicity over complexity.

**Deployment target:** Vercel (frontend + server functions) + Supabase cloud (Postgres + auth + realtime).

## Pattern Overview

**Overall:** Full-stack SSR/SPA hybrid with a managed backend

**Key Characteristics:**
- TanStack Start handles routing, SSR, and server functions in one framework
- Supabase provides the entire backend layer: Postgres, auth sessions, realtime subscriptions, and edge functions
- No custom API server — server-side logic lives in TanStack Start `createServerFn` handlers
- Client subscribes to Supabase Realtime channels for live board updates
- Push notifications are triggered server-side via a Supabase Database Webhook → Edge Function pipeline

## Layers

**Routing & Pages:**
- Purpose: File-based routes defining pages and layouts
- Location: `src/routes/` (planned)
- Contains: Route files, layouts, loaders, page components
- Depends on: Components, server functions, Supabase client
- Used by: End users via browser navigation

**Server Functions:**
- Purpose: Type-safe server-side operations (data fetching, AI proxying, write operations)
- Location: `src/server/` or co-located in route files via `createServerFn` (planned)
- Contains: Task CRUD handlers, screenshot OCR proxy to Claude API, push subscription management
- Depends on: Supabase service-role client, Claude API client, VAPID keys
- Used by: Client components calling server functions directly

**Components:**
- Purpose: Reusable UI — task cards, forms, board layout, notification permission prompt
- Location: `src/components/` (planned)
- Contains: TaskCard, TaskForm, TaskBoard, ScreenshotUpload, StatusBadge, NotificationPrompt
- Depends on: Tailwind CSS, Supabase realtime subscriptions (via hooks)
- Used by: Route page components

**Supabase Client Layer:**
- Purpose: Typed database client, auth helpers, realtime subscriptions
- Location: `src/lib/supabase/` (planned)
- Contains: Client singleton, type definitions generated from schema, auth helpers
- Depends on: `@supabase/supabase-js`
- Used by: Server functions (service-role client), components (anon key client)

**PWA / Service Worker:**
- Purpose: Offline shell, caching strategy, push notification delivery
- Location: `public/sw.js` (generated by Workbox / vite-plugin-pwa) (planned)
- Contains: Cache strategies, push event handler
- Depends on: vite-plugin-pwa, Workbox
- Used by: Browser, OS push service

## Data Flow

**Manual Task Creation:**

1. Staff fills the task creation form in the browser
2. Component calls a `createServerFn` handler on submit
3. Server function validates input and inserts a row into `tasks` table via Supabase service-role client
4. Supabase Realtime broadcasts the INSERT event to all subscribed clients
5. Each connected browser receives the event and updates the board in place — no refresh needed
6. Simultaneously, a Supabase Database Webhook fires on `tasks` INSERT
7. Webhook triggers a Supabase Edge Function that sends Web Push to all stored push subscriptions

**Screenshot-to-Task Flow (Phase 3):**

1. Staff uploads or drags a KakaoTalk screenshot onto the upload UI
2. Component calls a `createServerFn` OCR handler
3. Server function base64-encodes the image and sends it to Claude vision API with an extraction prompt
4. Claude returns structured JSON: client name, phone, service, date/time, notes, request type, and confidence flags
5. Server function returns extracted fields to the client
6. Task creation form pre-fills with extracted values; low-confidence fields are visually highlighted
7. Staff reviews, corrects if needed, and submits — continuing the normal task creation flow

**Realtime Board Sync:**

1. On mount, the TaskBoard component subscribes to a Supabase Realtime channel for the `tasks` table
2. Supabase streams INSERT, UPDATE, and DELETE events to all connected clients
3. Client-side state is updated directly from the realtime payload — no re-fetch needed
4. Subscription is torn down on component unmount

**Push Notification Flow:**

1. Staff grants push permission via the in-app NotificationPrompt
2. Browser generates a push subscription object (endpoint + keys)
3. Subscription is POSTed to a server function and stored in a `push_subscriptions` table
4. On `tasks` INSERT, the Supabase Database Webhook calls an Edge Function
5. Edge Function fetches all rows from `push_subscriptions` and sends a Web Push message to each endpoint using the VAPID key pair
6. Device OS delivers the notification even if the PWA is not open

**Authentication Flow:**

1. Staff visits the app; unauthenticated users are redirected to `/login`
2. Login form calls Supabase Auth `signInWithPassword`
3. Supabase issues a JWT session stored in a cookie (handled by `@supabase/ssr`)
4. Session persists across refreshes; server functions validate the session cookie before executing

## Key Abstractions

**Task:**
- Purpose: The core data entity representing a logged call lead
- Fields: id, client_name, phone, service, preferred_datetime, notes, request_type (enum), status (Open/In Progress/Done), created_by, updated_by, created_at, updated_at
- Table: `tasks` in Postgres
- RLS: All authenticated staff can read/write all tasks (flat team model)

**Push Subscription:**
- Purpose: Stores browser push endpoints so the Edge Function can fan out notifications
- Table: `push_subscriptions` in Postgres
- Fields: id, staff_id (FK to auth.users), endpoint, p256dh, auth, created_at

**Server Function (createServerFn):**
- Purpose: TanStack Start's typed RPC mechanism — runs on the server, callable from client components
- Used for: Task mutations, OCR processing (to keep the Claude API key server-side), push subscription storage

## Entry Points

**Application Root:**
- Location: `src/routes/__root.tsx` (planned)
- Triggers: Every page load
- Responsibilities: Global layout, auth session provider, Supabase client initialization

**Login Page:**
- Location: `src/routes/login.tsx` (planned)

**Task Board (Home):**
- Location: `src/routes/index.tsx` (planned)
- Triggers: Authenticated staff navigating to `/`
- Responsibilities: Load initial tasks, mount realtime subscription, render board

**Screenshot OCR Handler:**
- Location: `src/server/ocr.ts` or co-located in the upload route (planned)
- Triggers: Staff submitting a screenshot upload
- Responsibilities: Proxy image to Claude API, return extracted fields

**Push Notification Edge Function:**
- Location: `supabase/functions/notify-task/index.ts` (planned)
- Triggers: Supabase Database Webhook on `tasks` INSERT
- Responsibilities: Fan out Web Push to all stored subscriptions

## Error Handling

**Strategy:** Fail fast on the server, degrade gracefully on the client

**Patterns:**
- Server functions throw typed errors; TanStack Start surfaces them to the calling component
- Screenshot OCR failures fall back to an empty manual entry form with a user-facing message
- Realtime disconnections are handled by Supabase client reconnect logic; board shows a stale-data indicator
- Push notification delivery failures are logged but do not block task creation
- Form validation errors are shown inline before any server call is made

## Cross-Cutting Concerns

**Auth:** Supabase Auth with `@supabase/ssr` cookie-based session. Server functions read the session to identify the acting user for `updated_by` / `created_by` fields.

**Validation:** Client-side form validation before submission (required fields: client name, phone). Server functions re-validate inputs before writing to the database.

**RLS:** Supabase Row Level Security enforces that only authenticated users can read or write `tasks` and `push_subscriptions`. All staff have equal access (no role hierarchy).

**Styling:** Tailwind CSS with a mobile-first approach. Breakpoints: 375px (mobile baseline), 1024px (desktop layout).
